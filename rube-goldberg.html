Robotic Rube Goldberg Machine

The Challenge

As part of the University of Pittsburgh's Foundations of Engineering Design course, our team was tasked with designing and building a multi-stage Rube Goldberg machine, a deliberately complex chain-reaction device engineered to complete a simple task through a sequence of interconnected steps. The twist: at least one stage had to be robotic, incorporating a programmed microcontroller that responded to a real-world sensor input.

The challenge was not just mechanical. It required integrating embedded software with physical hardware, ensuring reliable sensor thresholds, and designing timing sequences precise enough to trigger downstream stages without failure.

Design Constraint: The robotic stage had to use analog sensor input to trigger an output in real time, with no manual intervention once the sequence began.

Technical Implementation

The robotic stage used an Arduino Uno microcontroller programmed in C++ to read analog input from a photoresistor. When the preceding mechanical stage lit a flame in the sensor's field of view, the produced light increased the analog reading above our threshold, and the Arduino triggered a piezo buzzer to play the Star Wars Imperial March.

// Photoresistor pin and threshold const int LDR_PIN = A0; const int THRESHOLD = 400; // calibrated for ambient light conditions const int BUZZER_PIN = 8; // Note frequencies (Hz) int melody[] = {440, 440, 440, 349, 523, 440, 349, 523, 440}; int durations[] = {500, 500, 500, 350, 150, 500, 350, 150, 700}; void loop() { int ldrValue = analogRead(LDR_PIN); if (ldrValue < THRESHOLD) { playMelody(); // trigger on light interruption } }

Calibrating the threshold was one of the most critical steps. The photoresistor's analog reading varied depending on room lighting conditions, so the threshold was tuned during testing to reliably distinguish between ambient light and the shadow cast by the incoming mechanical element — without false triggers from normal lighting fluctuations.

Engineering Lessons

This project was my first experience integrating embedded software with a physical mechanical system, and it exposed a set of engineering challenges that purely software or purely mechanical projects don't present. The sensor had to be reliable enough that a brief shadow reliably triggered the melody without misfiring. The timing of the audio output had to be short enough that the downstream stage received its cue before the mechanical momentum dissipated.

It reinforced a core principle of systems engineering: the interface between subsystems is often where designs fail. Writing code that works in isolation is one problem; writing code that works as one node in a physical chain-reaction system is another.
